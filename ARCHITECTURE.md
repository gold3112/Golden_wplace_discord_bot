# Golden Wplace Discord Bot アーキテクチャ

このドキュメントでは、ボットの内部構造、設計思想、およびコントリビューター向けの技術解説を行います。

## プロジェクト構造

```text
cmd/bot/
  └── main.go              # エントリーポイント。初期化とシグナルハンドリング。
internal/
  ├── commands/            # Discordコマンド（スラッシュ、ボタン、モーダル）の実装。
  ├── watchmanager/        # 監視エンジン。スケジューリングと差分計算の調整。
  ├── wplace/              # Wplace APIとの通信、タイル取得、パレット管理。
  ├── storage/             # JSONベースのデータ永続化。アトミック書き込み。
  ├── notifications/       # DiscordへのEmbed通知とTier計算。
  ├── models/              # アプリケーション全体で共有されるデータ構造。
  └── utils/               # 座標変換、レートリミッター、ID生成などの汎用ツール。
data/                      # 監視設定やテンプレート画像が保存されるディレクトリ。
```

## 主要コンポーネントの動作

### 1. 監視エンジン (`watchmanager`)
各監視タスクは個別のゴルーチン（`watchTask`）として実行されます。
- **スケジューリング**: デフォルト5分間隔にランダムなジッターを加え、API負荷を分散。
- **排他制御**: `Manager` は Watch ID ごとの Mutex を持ち、定期実行と手動実行 (`/watch now`) が重なった場合の競合を防止。
- **起動時整合性**: 起動時に Discord API を呼び出し、削除されたチャンネルに紐づく古い監視データを自動クリーンアップ。

### 2. 画像処理と差分検知 (`wplace`, `watchmanager/diff.go`)
- **アルファマスクの活用**: テンプレート画像の透明ピクセルを無視し、描画されている部分のみを比較。
- **パレット補正**: ユーザーがパレット外の色を含む画像を上げた場合、ユークリッド距離を用いてWplaceの公式パレット色へ近似。
- **タイル合成**: 監視エリアが複数のWplaceタイル（1000x1000）にまたがる場合、必要なタイルのみを取得し、メモリ上で1枚の画像に合成。

### 3. データ永続化 (`storage`)
- **アトミック書き込み**: 一時ファイルに書き込んでからリネームすることで、書き込み中のクラッシュによるデータ破損を防止。
- **マルチサーバー対応**: ギルドIDごとにディレクトリを分離。

### 4. Discord ユーザーフロー
- **インタラクティブ性**: ボタンとモーダルを多用し、チャットコマンドの記憶負担を軽減。
- **権限管理**: ボットが作成したチャンネルか、既存のチャンネルかを `IsExternalChannel` フラグで管理し、削除時の挙動（チャンネルを消すか残すか）を適切に切り替え。

## 並行性モデルと安全性

- **レート制限**: `utils.RateLimiter` により、サーバー（ドメイン）ごとに秒間のリクエスト数を制御。Wplace APIへの礼儀正しいアクセスを維持。
- **メモリ管理**: 監視の一時停止や削除時に、メモリ上の通知状態（`notifyStates`）も掃除。
- **スレッドセーフな更新**: `Storage.UpdateWatch` は読み込みから書き込みまで一貫してロックを保持し、Lost Update 問題を回避。

## 開発のガイドライン

1.  **UIの整合性**: スラッシュコマンドを追加する際は、可能な限り `/watch settings` のようにボタンによる後続操作を提供してください。
2.  **APIへの負荷**: タイル取得数を最小限にするため、座標計算には `utils.CountRequiredTiles` を必ず使用し、制限（最大8枚）を遵守してください。
3.  **エラーフィードバック**: ユーザー起因のエラー（不正な座標など）は `respondEphemeral` を使い、即座にフィードバックを返してください。
